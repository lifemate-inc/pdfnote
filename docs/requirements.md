# Smart PDF 要件定義書

**バージョン:** 0.1（Phase 0 ドラフト）
**最終更新:** 2026-05-23
**ステータス:** Phase 0 設計フェーズ - ユーザーレビュー前

---

## 1. プロダクト概要

### 1.1 目的
事業所での書類業務、特に「100ページ超の大量 PDF を分割・整理・メモ追記する作業」を効率化する、軽量・無料・ローカル完結のブラウザ完結型 PDF 編集ツール。

### 1.2 開発の背景
- Adobe Acrobat は分割・抽出・回転が有料版限定で、中小事業所には機能過多・コスト高
- 介護事業所では「120ページ・100人分の書類が1つのPDFにまとめられて送られてくる」業務が日常的に発生
- 同様のニーズは医療・士業・教育・自治体・中小企業全般に存在

### 1.3 プロダクトのポジショニング
- **介護事業所をビーチヘッド（最初の顧客）として開発**
- 設計は業界中立にし、全業界に展開可能とする
- 機能数で勝負せず、「シンプルさ」「安全性」「無料」で差別化

---

## 2. 目的・目標

### 2.1 ビジネス目標
1. **Phase 3 までに**介護事業所 5 件以上で実運用される
2. **Phase 4 までに**他業種（医療・士業など）の利用者を獲得する
3. 継続運用コストを最小化する（GitHub Pages 配信・OSS コミュニティ）

### 2.2 プロダクト目標
1. IT に不慣れな利用者が **5 分以内に基本操作を習得**できる
2. **120 ページの PDF を実用速度（30秒以内に一覧表示）**で扱える
3. **個人情報がインターネットに出ない**ことを技術的に保証する

---

## 3. ターゲットユーザー（ペルソナ）

### 3.1 ペルソナ A: 介護事業所の事務スタッフ（主要ターゲット）
- **属性:** 40〜60代の女性事務担当、ITは「メールと請求ソフトが使える」レベル
- **業務環境:** Windows 10/11 ノートPC（13-15インチ、メモリ 4-8GB）
- **典型業務:** ケアマネジャーから送られてくる「100人分の利用票がまとまった120ページPDF」を、人ごとに分割して各担当者へ配布
- **困りごと:**
  - 現状は紙に印刷して仕分け、または Acrobat 有料版を使用
  - 印刷代・トナー代がかさむ
  - Acrobat は機能が多すぎてどこを押せばいいか分からない
  - 一覧表示が小さくて見づらい

### 3.2 ペルソナ B: 士業の事務員（拡張ターゲット）
- **属性:** 30〜50代、PC操作は中程度
- **業務環境:** Windows / Mac 両方の事業所が存在
- **典型業務:** クライアントから受領した大量の書類PDFを、案件・期日別に整理・リネーム
- **困りごと:** ファイル名の一括変更・整理に手間がかかる

### 3.3 ペルソナ C: 医療機関の医事課職員（拡張ターゲット）
- **属性:** 30〜50代、PC操作は基本レベル
- **業務環境:** Windows 中心
- **典型業務:** 患者ごとに分かれていない PDF を分割・電子カルテに紐付け
- **困りごと:** 個人情報を扱うため、クラウドサービスは使えない

## 4. ユーザーストーリー

### US-1: 一覧で全体を把握する
> 介護事業所スタッフとして、120ページのPDFを開いたとき、全ページのサムネイル一覧を一目で見たい。なぜなら、どこで分割すべきかを視覚的に判断する必要があるから。

### US-2: 人ごとに分割する
> 介護事業所スタッフとして、PDFの中の「○○さん」の書類部分（例: 3〜5ページ）を選んで、その人の名前のPDFファイルとして保存したい。

### US-3: 分割後すぐにリネームする
> 介護事業所スタッフとして、抽出したPDFのプレビューを見ながら、「○○さん 2026年5月分」のような名前にすぐに変更したい。鉛筆アイコンをクリックするだけで編集できると嬉しい。

### US-4: 横向きページを直す
> 介護事業所スタッフとして、横向きで読みづらいページを一覧画面から1クリックで回転させたい。

### US-5: メモを残す
> 介護事業所スタッフとして、PDFの上から「確認済」「要対応」などの文字をスタンプのように貼り付けたい。元のPDFの文字は変更したくない。

### US-6: 重いPDFでも作業を進める
> 低スペックPCを使うスタッフとして、120ページのPDFが重くて開けない場合に、「最初に50ページずつに分割して保存」してから、軽くなったファイルで作業を続けたい。

### US-7: 安心して使う
> 介護事業所の責任者として、利用者の個人情報を扱うため、PDFデータがインターネットに送信されないことを確信して使いたい。

### US-8: 困ったときに助けを得る
> 介護事業所スタッフとして、操作が分からないときに、5分以内の動画や画面内ガイドで解決したい。

### US-9: 不具合を簡単に報告する
> 介護事業所スタッフとして、アプリの不具合に遭遇したとき、ボタン1つで開発者に報告できると嬉しい（個人情報は含まれないと安心）。

## 5. 機能要件

### 5.1 PDF読み込み
| ID | 要件 |
|----|------|
| F-1.1 | ドラッグ&ドロップで PDF を読み込める |
| F-1.2 | 「ファイルを選択」ボタンからも読み込める |
| F-1.3 | 読み込み中はプログレスバーで進捗を表示 |
| F-1.4 | 読み込み時にファイルサイズ・ページ数を検出し、重い場合は警告 |
| F-1.5 | パスワード保護 PDF は「対応していません」と明示してエラー終了 |

### 5.2 一覧表示
| ID | 要件 |
|----|------|
| F-2.1 | 全ページのサムネイルをグリッド表示する |
| F-2.2 | サムネイルサイズをスライダーで変更できる（小・中・大） |
| F-2.3 | ノートPC画面（13-15インチ）で 15-20 ページが同時表示される |
| F-2.4 | 各サムネイルに**ページ番号**を表示 |
| F-2.5 | 各サムネイルに「選択チェックボックス」「回転ボタン」を表示 |
| F-2.6 | サムネイルクリックでプレビュー画面に遷移 |
| F-2.7 | 連続範囲選択（Shift+クリック）、複数選択（Ctrl+クリック）対応 |

### 5.3 分割・抽出
| ID | 要件 |
|----|------|
| F-3.1 | 一覧で選択したページを「新規PDF」として抽出できる |
| F-3.2 | 抽出時にファイル名を入力する（後で変更可） |
| F-3.3 | 抽出は元PDFを変更しない（非破壊操作） |
| F-3.4 | 複数の抽出を連続して行える（残ったページは保持） |
| F-3.5 | 抽出履歴を「処理済みファイル一覧」として画面下部に表示 |

### 5.4 事前分割（軽量化モード）
| ID | 要件 |
|----|------|
| F-4.1 | 「N ページずつに自動分割」ボタンをトップ画面に配置 |
| F-4.2 | N の値はデフォルト 50、ユーザーが変更可能 |
| F-4.3 | 分割後のファイル名は「元ファイル名_part1.pdf」「元ファイル名_part2.pdf」のように自動生成 |
| F-4.4 | 分割処理はストリーミングで実装し、メモリ消費を最小化 |
| F-4.5 | 進捗バーで処理状況を表示 |
| F-4.6 | 重い PDF を検出した際に「軽量化を推奨」のガイドを自動表示 |

### 5.5 ページ回転
| ID | 要件 |
|----|------|
| F-5.1 | 一覧のサムネイルから 1 クリックで時計回り 90° 回転 |
| F-5.2 | プレビュー画面でも回転ボタンが利用できる |
| F-5.3 | 複数選択中のページを一括回転できる |
| F-5.4 | 回転は元 PDF を変更しない（出力時に反映） |

### 5.6 リネーム
| ID | 要件 |
|----|------|
| F-6.1 | プレビュー画面の上部にファイル名を表示 |
| F-6.2 | ファイル名の横に**鉛筆アイコン**を配置 |
| F-6.3 | 鉛筆アイコンクリックでインライン編集モードになる |
| F-6.4 | Enter キーまたは外側クリックで確定 |
| F-6.5 | Esc キーでキャンセル |
| F-6.6 | リネーム結果はダウンロード時のファイル名に反映 |

### 5.7 文字スタンプ
| ID | 要件 |
|----|------|
| F-7.1 | プレビュー画面で「スタンプ追加」ボタンをクリック |
| F-7.2 | PDF 上の任意の位置をクリックしてテキスト入力欄を表示 |
| F-7.3 | フォントサイズ（小・中・大）と色（黒・赤・青）を選択可 |
| F-7.4 | 追加したスタンプはドラッグで位置移動可 |
| F-7.5 | 追加したスタンプはクリックで削除可 |
| F-7.6 | スタンプは元 PDF の文字を変更しない（オーバーレイ） |
| F-7.7 | 出力時に PDF にフラット化して埋め込み |

### 5.8 ダウンロード
| ID | 要件 |
|----|------|
| F-8.1 | 「保存」ボタンで編集結果を PDF として出力 |
| F-8.2 | File System Access API 対応ブラウザでは保存先を指定可 |
| F-8.3 | 非対応ブラウザ（Safari / Firefox）ではダウンロードフォルダへ出力 |
| F-8.4 | 出力ファイル名はリネーム結果を反映 |

### 5.9 自動保存
| ID | 要件 |
|----|------|
| F-9.1 | 編集中の状態（選択・回転・スタンプ・リネーム）を IndexedDB に自動保存 |
| F-9.2 | ブラウザを閉じて再度開いたとき、「前回の作業を再開しますか？」と表示 |
| F-9.3 | 復元時は元 PDF の再選択を求める（PDF データ自体は保存しない＝個人情報保護） |
| F-9.4 | 30 日後に自動保存データを削除 |

### 5.10 不具合報告
| ID | 要件 |
|----|------|
| F-10.1 | 画面右下に「不具合報告」ボタンを配置 |
| F-10.2 | クリックで報告フォームを表示（症状・再現手順を記入） |
| F-10.3 | PDF ファイル名・内容は**絶対に**含めない |
| F-10.4 | ブラウザバージョン・OS・スタックトレースのみを送信 |
| F-10.5 | 送信先は GitHub Issue 自動起票 or サポートメール（Phase 0 後半で決定） |

### 5.11 初回ガイドツアー
| ID | 要件 |
|----|------|
| F-11.1 | 初回起動時に 5 ステップ程度のガイドツアーを表示 |
| F-11.2 | 各機能の位置と簡単な使い方を説明 |
| F-11.3 | スキップ可能、いつでも再表示可能 |

### 5.12 ヘルプ・チュートリアル
| ID | 要件 |
|----|------|
| F-12.1 | 画面右上に「ヘルプ」ボタンを配置 |
| F-12.2 | クリックで FAQ ページとチュートリアル動画へのリンクを表示 |
| F-12.3 | 「重い PDF の扱い方」など運用ベストプラクティスを掲載 |

### 5.13 ブラウザ互換性チェック
| ID | 要件 |
|----|------|
| F-13.1 | 起動時にブラウザバージョンを検出 |
| F-13.2 | 非推奨ブラウザでは警告バナーを表示 |
| F-13.3 | 完全非対応の機能（File System Access API 等）はフォールバック動作 |

## 6. 非機能要件

### 6.1 性能
| ID | 要件 | 目標値 |
|----|------|--------|
| N-1.1 | 120 ページ PDF の読み込み時間 | 30 秒以内 |
| N-1.2 | 抽出処理時間（5 ページあたり） | 5 秒以内 |
| N-1.3 | メモリ使用量（120ページPDF） | 2 GB 以下 |
| N-1.4 | 起動時間 | 3 秒以内 |

### 6.2 セキュリティ
| ID | 要件 |
|----|------|
| N-2.1 | PDF データは**インターネットに送信しない** |
| N-2.2 | Service Worker で外部通信をホワイトリスト方式に制限 |
| N-2.3 | CSP (Content Security Policy) で外部スクリプト読み込みを禁止 |
| N-2.4 | エラー報告でも PDF ファイル名・内容を含めない |
| N-2.5 | 自動保存は PDF データ自体を保存せず、メタ情報（選択状態等）のみ |

### 6.3 互換性
| ID | 要件 |
|----|------|
| N-3.1 | Windows 10 / 11、macOS 最新版で動作 |
| N-3.2 | Chrome / Edge 最新版で完全動作 |
| N-3.3 | Safari / Firefox では機能制限ありで動作 |
| N-3.4 | スクリーンリーダー対応（WCAG 2.1 AA レベル） |

### 6.4 ユーザビリティ
| ID | 要件 |
|----|------|
| N-4.1 | フォントサイズ最小 16px |
| N-4.2 | カラーコントラスト比 4.5:1 以上 |
| N-4.3 | キーボード操作完結（マウスなしでも基本操作可） |
| N-4.4 | 操作ボタンは最小 44×44px |
| N-4.5 | エラーメッセージは日本語・平易な表現 |

### 6.5 保守性
| ID | 要件 |
|----|------|
| N-5.1 | TypeScript で型安全実装 |
| N-5.2 | ESLint / Prettier でコード品質統一 |
| N-5.3 | 主要機能にユニットテストを配置 |
| N-5.4 | GitHub Actions で CI/CD 自動化 |

### 6.6 配布・運用
| ID | 要件 |
|----|------|
| N-6.1 | GitHub Pages で配信 |
| N-6.2 | Service Worker で自動更新（透過的） |
| N-6.3 | 旧バージョンを `/v1.0.0/` 形式で保持 |
| N-6.4 | リリースノートを GitHub Releases に掲載 |

## 7. スコープ外（やらないこと）

| 項目 | 理由 |
|------|------|
| PDF 内既存テキストの編集 | スタンプ上書きで代替。本格的なテキスト編集は複雑度が高すぎる |
| OCR（画像 PDF の文字認識） | 専用ライブラリ必要で配布サイズ増大 |
| フォーム入力機能 | 業務によって要件が大きく異なるため |
| パスワード保護 PDF の解除 | セキュリティ的に敏感 |
| 電子署名 | 姉妹プロジェクト [hpki-signer](https://github.com/lifemate-inc/hpki-signer) で対応 |
| 複数 PDF の結合 | スコープ拡大防止（Phase 2 以降で再検討） |
| クラウド保存・同期 | ローカル完結の原則を守る |
| スマートフォン対応 | ノートPC最適化に集中 |
| 多言語対応 | まず日本語に集中、Phase 4 以降で検討 |

## 8. 制約事項

### 8.1 技術的制約
- ブラウザの File API のサイズ制限に依存
- IndexedDB の容量制限（おおむね数百MB〜数GB、ブラウザ依存）
- WebWorker 経由でも処理性能はブラウザ・PC スペックに依存

### 8.2 法的・倫理的制約
- 個人情報保護法を順守する設計
- 利用者が PDF を取り扱う際の責任は利用者に帰属（MIT ライセンスで明示）

### 8.3 リソース制約
- 開発実行: Claude
- 意思決定: ユーザー
- パイロット協力: 既存業務関係先 1-2 社（予定）
